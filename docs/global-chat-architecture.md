# خطة بناء منصة دردشة عالمية

هذه الوثيقة تثبت القرار الهندسي الأول لهذا المشروع حتى لا نكرر مشاكل المشاريع السابقة: لا نبني ميزات دردشة فوق بيانات وهمية أو منطق محلي داخل الواجهة فقط، ولا نضيف تقنيات كثيرة قبل أن يكون مسار الرسائل والمصادقة والتوسع واضحاً.

## الوضع الحالي

- الواجهة مبنية بـ React وTypeScript وVite، وهذا مناسب كبداية قوية لواجهة دردشة حديثة.
- يوجد Nitro server layer داخل `server/`، لكنه ليس كافياً بعد كطبقة دردشة عالمية.
- صفحة الغرفة الحالية تعرض رسائل محلية داخل React state، لذلك هي نموذج واجهة وليست دردشة حقيقية متعددة المستخدمين.
- تسجيل الدخول التجريبي كان يستخدم مساراً غير آمن ومكسور البناء، وتم تعطيله مؤقتاً حتى لا نعتمد على كلمة مرور نصية أو قاعدة بيانات غير جاهزة.
- توجد مجلدات `apps/api` و`apps/web` غير متتبعة حالياً وتبدو كقوالب منفصلة. لا يجب خلطها مع الجذر قبل اتخاذ قرار monorepo واضح.

## القرار المعماري المقترح

المرحلة العملية الأفضل لهذا المشروع:

1. إبقاء الواجهة الحالية في الجذر مؤقتاً: React + TypeScript + Tailwind.
2. استخدام API مستقل لاحقاً للدردشة الحقيقية، والأفضل أن يكون Node.js/TypeScript في البداية مع WebSocket.
3. استخدام PostgreSQL للبيانات الدائمة مثل المستخدمين، ملفات الأعضاء، الغرف، الصلاحيات، المحافظ، والهدايا.
4. استخدام Redis للحضور اللحظي، الجلسات السريعة، rate limits، وتوزيع الرسائل بين أكثر من سيرفر.
5. عند الوصول إلى حجم كبير جداً، إضافة message broker مثل NATS أو Kafka لمسارات الأحداث الثقيلة.

## مسار الرسائل الإنتاجي

المسار الصحيح للرسالة يجب أن يكون كالتالي:

1. المستخدم يرسل الرسالة من الواجهة.
2. الواجهة تنشئ realارسال حقيقي واستبقال حقيقي من السيرفر echo ثابت الشكل حتى لا يحدث اهتزاز في واجهة الجوال.
3. WebSocket server يتحقق من الجلسة والصلاحيات وحجم الرسالة.
4. الرسالة تحفظ في PostgreSQL.
5. الرسالة تنشر عبر Redis Pub/Sub أو broker إلى كل سيرفرات الدردشة.
6. كل سيرفر يبث الرسالة للمستخدمين الموجودين في الغرفة.
7. الواجهة تستبدل local echo بالرسالة المؤكدة بدون تغيير شكل الفقاعة أو أزرار الرد والحذف.

## قواعد يجب عدم كسرها

- النصوص العربية تحفظ دائماً بترميز UTF-8.
- نستفيد من الشات القديم كمرجع للخصائص والسلوك المتوقع فقط، ولا ننسخ مسمياته أو طريقة تكويده.
- لا نعرض أرقام متصلين وهمية في النسخة الإنتاجية.
- لا نستخدم `localStorage` كمصدر ثقة للجلسة أو الصلاحيات.
- لا نقارن كلمات المرور كنص صريح. يجب استخدام Argon2 أو bcrypt.
- لا نخلط بين mock data وبيانات الإنتاج في نفس المسار.
- لا نضيف WebSocket قبل وجود نموذج رسائل وغرف واضح.
- لا نغيّر بنية الواجهة بالكامل دفعة واحدة؛ نثبت المسار ثم ننقل المكونات تدريجياً إلى بيانات حقيقية.

## نموذج قاعدة البيانات الأولي

الجداول الأساسية:

- `users`: الحسابات، الاسم، البريد أو الهاتف، كلمة المرور المشفرة، الحالة، الدور.
- `sessions`: الجلسات النشطة وتجديد الرموز.
- `rooms`: الغرف، النوع، المالك، الإعدادات، حالة الغرفة.
- `room_members`: العضويات والصلاحيات داخل الغرف.
- لا نستخدم جدول رسائل دائم لرسائل الغرف أو الخاص حسب قرار المشروع. أي جدول رسائل موجود في المخطط هو مرشح للإزالة أو للاستخدام فقط في الاستثناءات الإدارية مثل البلاغات/التثبيت.
- `moderation_events`: الحظر، الكتم، الحذف، تقارير الإساءة.

## مراحل التنفيذ

### المرحلة 1: تثبيت الأساس

- إصلاح البناء.
- إصلاح النصوص العربية المتضررة في الصفحات الأساسية.
- إزالة الاعتماد على أرقام ورسائل وهمية من الأماكن التي ستتحول إلى بيانات حقيقية.
- إضافة health API وفصل عقود API عن الواجهة.

### المرحلة 2: مصادقة آمنة

- تسجيل زائر وعضو.
- تشفير كلمات المرور.
- جلسات آمنة.
- صلاحيات واضحة: زائر، عضو، مشرف، مالك، مدير.

### المرحلة 3: دردشة حقيقية

- WebSocket endpoint.
- دخول وخروج الغرف.
- إرسال واستقبال الرسائل.
- حفظ الرسائل في قاعدة البيانات.
- حضور online في Redis.

### المرحلة 4: قابلية التوسع

- Redis adapter أو Pub/Sub.
- rate limiting.
- تقسيم الغرف الكبيرة.
- مراقبة الأداء والأخطاء.
- اختبارات ضغط.

### المرحلة 5: ميزات اجتماعية

- الخاصة.
- الحائط.
- القصص.
- الهدايا.
- البوتات.
- أدوات الإدارة والمراقبة.

## معيار النجاح للمرحلة القادمة

المرحلة القادمة لا تعتبر مكتملة إلا إذا:

- البناء ينجح.
- النصوص العربية الأساسية سليمة.
- توجد عقود واضحة للرسائل والغرف.
- أول مسار رسالة لا يعتمد على mock data.
- واجهة الجوال لا يحدث فيها تغير شكل عند إرسال الرسالة ثم تأكيدها من السيرفر.

## حالة التنفيذ الحالية

- تم إنشاء أول مسار إرسال واستقبال حقيقي عبر Nitro API:
  - `GET /api/rooms`
  - `GET /api/rooms/:roomId/messages`
  - `POST /api/rooms/:roomId/messages`
  - `GET /api/rooms/:roomId/stream`
- الاستقبال الحالي يستخدم Server-Sent Events حتى تصل الرسائل بسرعة للواجهة بدون polling مستمر.
- التخزين الحالي داخل ذاكرة السيرفر فقط، وهو مناسب للتطوير والتحقق من تدفق الواجهة، لكنه ليس مناسباً للإنتاج.
- خطوة الإنتاج التالية هي نقل التخزين إلى PostgreSQL ونقل البث بين السيرفرات إلى Redis Pub/Sub أو WebSocket adapter.
- عند استهداف ملايين المستخدمين، يجب ألا نعتمد على ذاكرة سيرفر واحد؛ كل الرسائل والحضور والجلسات يجب أن تكون قابلة للتوزيع أفقياً.
- تم تجهيز حفظ الحسابات الدائمة في PostgreSQL عبر جدول `users` وجدول `user_profiles`، مع fallback مؤقت للذاكرة عند غياب قاعدة البيانات في بيئة التطوير.
