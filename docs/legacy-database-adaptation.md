# الاستفادة من قاعدة Safari القديمة

قاعدة `safarihost` القديمة مفيدة جداً كمصدر لفهم المنتج، لكنها ليست مناسبة كتصميم نهائي لمنصة دردشة عالمية. سنستفيد منها على مستوى الكيانات والقواعد، لا على مستوى النسخ الحرفي.

## قاعدة ثابتة للتطوير

لا ننسخ المشروع القديم في المسميات أو طريقة التكويد أو بنية الجداول. المشروع القديم يستخدم فقط كمرجع لمعرفة الخصائص التي يتوقعها المستخدم داخل الشات. أي خاصية مهمة موجودة هناك يجب أن يكون لها مقابل حديث ومنظم في مشروعنا، لكن بأسماء واضحة وتصميم قابل للتوسع.

أمثلة على ذلك:

- البنر القديم يقابله في مشروعنا `profileBannerUrl`.
- ثيم العضو يقابله `profileThemeId`.
- إطار الصورة يقابله `avatarFrameUrl`.
- أيقونة الهدية تقابلها `giftIconUrl`.
- أيقونة/شارة الملف تقابلها `profileIconUrl`.
- زخرفة الاسم تقابلها `nameGradient` أو `nameEffectId`.
- شكل فقاعة الرسائل يقابله `messageBubbleStyle`.

بهذا نحافظ على اكتمال المنتج من ناحية الميزات، لكن بدون توريث ديون تقنية أو أسماء قديمة غير واضحة.

## ما سنأخذه منها

- `users`: مصدر لفهم ملف العضو، الألوان، الصورة، الرتبة، العملات، والتوثيق.
- `rooms`: مصدر لفهم الغرف، المايكات، الترحيب، مالك الغرفة، وإعدادات الغرفة.
- `bars`: مصدر لفهم الحائط والمنشورات والتعليقات والإعجابات.
- `bots`: مصدر لفهم البوتات ورسائل الترحيب.
- `powers`: مصدر لفهم الصلاحيات والرتب.
- `storie` و`storieview`: مصدر لفهم القصص والمشاهدات.
- `followers`: مصدر لفهم المتابعة.
- `coin_ledger`, `gift_notifications`, `site_master_wallet`, `agent_wallets`: مصدر لفهم المحفظة والهدايا والحركات المالية.
- `agents`, `topup_orders`, `agent_wallet_ledger`: مصدر لفهم نظام الوكلاء والشحن.
- `band`, `notext`, `logs`, `stats`: مصدر لفهم الحظر، الفلترة، وسجلات الإدارة.

## ما لن ننقله كما هو

- كلمات المرور القديمة `sha1$...` لن تستخدم في النظام الجديد. عند أول تسجيل دخول يجب إجبار المستخدم على إعادة تعيين كلمة المرور أو ترحيلها بطريقة آمنة.
- تخزين التعليقات والإعجابات كـ JSON داخل `bars.bcc` و`bars.likes` لن يستمر. في النظام الجديد التعليقات والردود والإعجابات ستكون جداول مستقلة.
- الاعتماد على `uid`, `lid`, `idreg` كسلاسل أساسية لن يكون أساس الهوية. سنحفظها كـ `legacy_id` أو `metadata` فقط عند الترحيل.
- أرقام IP وبيانات الأجهزة الموجودة في dump لا يجب نقلها لبيئة إنتاج جديدة إلا إذا كان هناك سبب قانوني واضح.
- بيانات البوتات الوهمية ذات IP عشوائي لا تعتبر بيانات إنتاجية.

## خريطة التحويل

| MariaDB القديم | PostgreSQL الجديد | الملاحظة |
| --- | --- | --- |
| `users` | `users`, `wallets`, `wallet_ledger` | نفصل الحساب عن المحفظة والحركات |
| `rooms` | `rooms`, `room_members` | نفصل عضوية الغرفة وصلاحياتها |
| `bars` | `wall_posts`, `wall_comments` | لا نضع التعليقات داخل JSON طويل |
| `followers` | `follows` | نفس الفكرة مع مفاتيح UUID |
| `storie` | `stories` | كل عنصر قصة يصبح صفاً واضحاً |
| `gifts_git` داخل `settings` | `gifts` | الهدايا تصبح جدولاً مستقلاً |
| `coin_ledger` | `wallet_ledger` | دفتر واحد للحركات المالية |
| `agent_wallets` | `wallets` | محفظة الوكيل نوع من المحافظ |
| `agent_wallet_ledger` | `wallet_ledger` | حركة مستقلة للوكيل |
| `agents` | `agents` | مع ربط اختياري بحساب مستخدم |
| `topup_orders` | لاحقاً `topup_orders` | تؤجل حتى ننفذ الشحن |
| `band` | `moderation_events` | الحظر يصبح حدثاً له مدة وسبب |
| `notext` | لاحقاً `moderation_filters` | فلترة الكلمات في جدول مستقل |
| `powers` | لاحقاً `roles` و`permissions` | الصلاحيات تحتاج تفكيكاً منظماً |

## قاعدة العمولة

نظام الوكلاء يجب أن يبقى مبنياً على دفتر حركات واضح:

- سعر الباقة لا يزاد بسبب عمولة الوكيل.
- قيمة الباقة وعمولة الوكيل مفهومان منفصلان.
- عند تنفيذ الشحن، تسجل حركة خصم أو تسوية من محفظة الموقع/الدردشة.
- تسجل حركة مستقلة إلى محفظة الوكيل حسب النسبة أو القاعدة المعتمدة.
- كل حركة لها `txid`, `before_balance`, `after_balance`, `reference_type`, و`reference_id`.

## ترتيب الترحيل المقترح

1. إنشاء الجداول الجديدة من `database/global-chat.postgres.sql`.
2. ترحيل المستخدمين بدون كلمات مرور صالحة، مع وضع علامة `requires_password_reset`.
3. ترحيل الغرف العامة والمالكين.
4. ترحيل الحائط كمنشورات وتعليقات منفصلة.
5. ترحيل الهدايا والعملات بعد تثبيت قواعد المحفظة.
6. ترحيل الوكلاء والشحن بعد اختبار دفتر الحركات.
7. ترك السجلات القديمة كأرشيف قراءة فقط عند الحاجة.
